Trust & Verification Levels

Not all proofs are created equal. A theorem checked by the Lean kernel is more trustworthy than one an LLM says "looks right." Deppy makes this distinction explicit with a four-level trust lattice that tracks exactly how much confidence you can place in each verified property.

The Four Verification Levels

Level Badge What It Means How You Get It
🟢 LEAN_VERIFIED LEAN_VERIFIED Machine-checked by Lean 4's type-theoretic kernel. The proof is a constructive term that has been independently verified. Translate to Lean, compile with lake build, kernel accepts.
🟡 Z3_PROVEN Z3_PROVEN Proved by the Z3 SMT solver with no admits. Z3 is a mature, widely-trusted solver — but it's not a constructive proof, and bugs in Z3 exist. Encode as SMT-LIB formula, Z3 returns unsat (negation is unsatisfiable).
🟠 LLM_JUDGED 🟠 LLM_JUDGED An LLM oracle says the property is correct with confidence ≥ the configured threshold (default: 0.9). Useful for natural-language specs and heuristic checks. Oracle query returns {"correct": true, "confidence": 0.95}.
🔴 UNTRUSTED UNTRUSTED No verification has been performed. The spec is assumed on faith — it might be wrong. This is the starting point for all new specs. Default. Any spec without a proof is UNTRUSTED.
The levels form a total order: LEAN_VERIFIED > Z3_PROVEN > LLM_JUDGED > UNTRUSTED. Each level strictly dominates the ones below it.

When Each Level Is Appropriate

🟢 LEAN_VERIFIED — The Gold Standard

Use when:

  • Safety-critical code (medical, financial, aerospace)
  • Core data structure invariants
  • Cryptographic protocol correctness
  • Foundational theorems your entire codebase depends on
# A property so important it needs Lean verification
@guarantee("sorted list is a permutation of input", trust="LEAN_VERIFIED")
def sort(xs: list[int]) -> list[int]:
    # This generates a Lean proof obligation:
    # theorem sort_correct : ∀ xs, Sorted (sort xs) ∧ Perm (sort xs) xs
    ...
LEAN_VERIFIED is expensive. The Python → Lean translation adds development time. Reserve it for properties that must be correct.

🟡 Z3_PROVEN — The Pragmatic Choice

Use when:

  • Arithmetic bounds and overflow checks
  • Array index safety
  • Precondition verification at call sites
  • Linear arithmetic properties
  • Most sidecar precondition proofs
# Z3 handles these naturally
@guarantee("index is within bounds", trust="Z3_PROVEN")
def safe_get(arr: list[int], i: int) -> int:
    assume(i >= 0 and i < len(arr))
    return arr[i]
    # Z3 checks: 0 ≤ i < len(arr) at every call site

🟠 LLM_JUDGED — The Frontier

Use when:

  • Natural-language specifications that can't be formalized easily
  • Heuristic correctness checks during development
  • Semantic properties ("this function returns user-friendly error messages")
  • Quick smoke-testing before investing in formal proofs
# Properties that resist formalization
@guarantee("error message is helpful and mentions the invalid field",
           trust="LLM_JUDGED", confidence=0.85)
def validate_config(config: dict) -> str | None:
    # Oracle checks: does the returned string mention the bad field?
    ...
LLM_JUDGED is inherently unreliable. The oracle can hallucinate. Use this level as a stepping stone — flag properties as LLM_JUDGED to remind yourself they need formal proofs later.

🔴 UNTRUSTED — The Starting Point

Use when:

  • You've just written a spec and haven't verified it yet
  • External dependencies with no sidecar
  • Properties you plan to verify but haven't gotten to
# Every spec starts here
@about("requests.get")
class RequestsGetSpec(ExternalSpec):
    trust_level = "UNTRUSTED"  # TODO: add real verification

    @assume
    def postcondition(self, url, result):
        return hasattr(result, 'status_code')

Trust Propagation

The most important rule in Deppy's trust system: trust flows downward. A function's trust level is bounded by the minimum trust of everything it depends on.

The Rule

If function A calls function B, then trust(A) ≤ trust(B).

Formally: trust(f) = min(own_proof_level(f), min(trust(g) for g in deps(f)))

Example: Trust Propagation Chain

# trust_example.py
import math
from decimal import Decimal

@guarantee("result is positive")
def compute_distance(x: float, y: float) -> float:
    # Calls math.sqrt — trust depends on math.deppy
    return math.sqrt(x * x + y * y)

@guarantee("normalized vector has magnitude 1")
def normalize(x: float, y: float) -> tuple[float, float]:
    # Calls compute_distance — trust ≤ trust(compute_distance)
    d = compute_distance(x, y)
    return (x / d, y / d)

@guarantee("angle is in [0, 2π)")
def angle_between(x1, y1, x2, y2) -> float:
    # Calls normalize — trust ≤ trust(normalize)
    nx1, ny1 = normalize(x1, y1)
    nx2, ny2 = normalize(x2, y2)
    return math.atan2(ny2 - ny1, nx2 - nx1) % (2 * math.pi)
$ deppy verify trust_example.py --show-trust

Trust Propagation Analysis
═══════════════════════════════════════════════

math.sqrt        → 🟡 Z3_PROVEN  (from sidecar)
math.atan2       → 🔴 UNTRUSTED  (no sidecar spec)

compute_distance → 🟡 Z3_PROVEN
  own proof:   🟡 Z3_PROVEN (x²+y² ≥ 0 proved by Z3)
  deps:        math.sqrt = 🟡 Z3_PROVEN
  result:      min(🟡, 🟡) = 🟡 Z3_PROVEN

normalize        → 🟡 Z3_PROVEN
  own proof:   🟡 Z3_PROVEN (d > 0 proved by Z3)
  deps:        compute_distance = 🟡 Z3_PROVEN
  result:      min(🟡, 🟡) = 🟡 Z3_PROVEN

angle_between    → 🔴 UNTRUSTED  ← limited by math.atan2!
  own proof:   🟡 Z3_PROVEN
  deps:        normalize = 🟡, math.atan2 = 🔴
  result:      min(🟡, 🟡, 🔴) = 🔴 UNTRUSTED

───────────────────────────────────────────────
⚠ angle_between is 🔴 UNTRUSTED because math.atan2 has no sidecar spec.
  Fix: add a spec for math.atan2 in sidecars/math.deppy
Trust propagation makes verification gaps visible. Even if your own code is perfectly verified, a single unspecified library call drags the entire call chain down to UNTRUSTED.

Gradual Verification

Gradual verification is the practice of incrementally raising your project's trust level over time. Deppy is designed for this workflow.

The Gradual Verification Ladder

1
Day 1: Everything is 🔴 UNTRUSTED. That's fine. Add deppy.toml and deppy verify to your CI. The report shows 0% coverage.
2
Week 1: Write sidecar specs for your most-used library functions. Add @guarantee to your critical functions. Coverage climbs to 30%. Many are 🟠 LLM_JUDGED.
3
Month 1: Replace LLM_JUDGED specs with Z3 proofs where possible. Add refinement types. Coverage hits 60%, most at 🟡 Z3_PROVEN.
4
Quarter 1: Translate core invariants to Lean for 🟢 LEAN_VERIFIED. Set CI gate: "no regressions below Z3_PROVEN for src/core/."
5
Ongoing: New code must meet the team's trust floor. Refactor UNTRUSTED modules when touching them ("boy scout rule" for verification).

CI Integration

Deppy integrates with CI/CD pipelines to enforce trust levels and track verification coverage over time.

GitHub Actions Example

# .github/workflows/verify.yml
name: Deppy Verification
on: [push, pull_request]

jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install Deppy run: pip install git+https://github.com/thehalleyyoung/deppy.git && pip install z3-solver

      - name: Run verification
        run: |
          deppy verify src/ \
            --sidecar-dir sidecars/ \
            --report json \
            --output verification-report.json

      - name: Check trust floor
        run: |
          deppy trust gate verification-report.json \
            --require "src/core/**:Z3_PROVEN" \
            --require "src/api/**:LLM_JUDGED" \
            --no-regression

      - name: Upload report
        uses: actions/upload-artifact@v4
        with:
          name: verification-report
          path: verification-report.json

      - name: Comment on PR
        if: github.event_name == 'pull_request'
        run: |
          deppy trust diff \
            verification-report.json \
            main-report.json \
            --format markdown > comment.md
          gh pr comment ${{ github.event.number }} \
            --body-file comment.md

The --no-regression Flag

The --no-regression flag ensures that trust levels never decrease. If a function was 🟡 Z3_PROVEN and a PR changes it to 🔴 UNTRUSTED, the CI gate fails: 

$ deppy trust gate report.json --no-regression

Trust Gate Check
═══════════════════════════════════════════════

✓ src/core/types.py:    🟢 → 🟢 (no change)

✓ src/api/handlers.py:  🟡 → 🟡 (no change)
✗ src/api/auth.py:      🟡 → 🔴 REGRESSION
  Function verify_token was Z3_PROVEN, now UNTRUSTED.
  Cause: new dependency on jwt.decode (no sidecar spec).

Fix: add a sidecar spec for jwt.decode in sidecars/jwt.deppy

The Trust Dashboard

Deppy includes a local web dashboard for visualizing your project's trust landscape: 

$ deppy dashboard --port 8080
Dashboard running at http://localhost:8080

The dashboard shows:

  • Module map — a treemap colored by trust level (green → red).
  • Dependency graph — edges colored by trust level of the target. Red edges show where untrusted deps drag down trust.
  • History — trust coverage over time (like a code coverage chart).
  • Hotspots — functions that would improve the most coverage if verified.
The "hotspots" view is particularly useful: it ranks unverified functions by how many other functions depend on them. Verifying a hotspot raises the trust of its entire dependency tree.

Comparison with F*'s --admit and sorry

Other verification systems have mechanisms for incomplete proofs:

Feature Deppy F* Lean 4
Skip a proof trust_level = "UNTRUSTED" admit() sorry
Track skipped proofs ✅ Trust report, CI gates ⚠️ Manual --admit_smt_queries ⚠️ #check @[sorry]
Gradual trust levels ✅ 4 levels, lattice ❌ Binary (proved/admit) ❌ Binary (proved/sorry)
Trust propagation ✅ Automatic ⚠️ Manual dependency analysis ⚠️ Manual
CI coverage tracking ✅ Built-in ❌ External tooling ❌ External tooling
Verify external libs ✅ Sidecar proofs ❌ Must modify source ❌ Must write bindings

F*: admit()

(* F*: binary skip — loses all info *)
let compute_hash (data: bytes) =
  admit ();  (* black hole — no tracking *)
  sha256 data

(* No way to know WHY it was admitted,
   or what depends on this admit.  *)

Deppy: Trust Levels

# Deppy: tracked, graded, reported
@guarantee("SHA-256 hash",
           trust="LLM_JUDGED")
def compute_hash(data: bytes):
    return sha256(data)

# Trust report shows: LLM_JUDGED
# Dependents are bounded by 🟠
# CI tracks it for future upgrade

Configuring Trust Levels

# deppy.toml — trust configuration
[trust]
# Default trust level for new specs
default_level = "UNTRUSTED"

# LLM oracle settings
[trust.llm]
confidence_threshold = 0.9     # minimum confidence for LLM_JUDGED
model = "gpt-4"                # oracle model
temperature = 0.0              # deterministic for reproducibility
max_retries = 3                # retry on low confidence

# CI gates
[trust.gates]
# Minimum trust level by path glob
"src/core/**" = "Z3_PROVEN"
"src/api/**" = "LLM_JUDGED"
"src/utils/**" = "UNTRUSTED"    # no requirement (yet)

# Prevent trust regressions
no_regression = true

# Fail CI if overall coverage drops below this
min_coverage = 0.70

  • log() is 🟠 LLM_JUDGED
