Getting Started with SynHoPy

SynHoPy verifies existing Python code — you do not need to rewrite anything or learn a new language. You keep writing ordinary Python with standard type hints, add lightweight specification decorators, and SynHoPy checks that the code actually satisfies those specifications using Z3, cubical homotopy type theory, and optional sidecar proofs.

This chapter walks you through installation, your first verified function, and how to read the verifier's output.

Python version. SynHoPy requires Python 3.10 or later. It relies on match statements, ParamSpec, and several newer typing features. We recommend Python 3.12+ for the best experience.

Installation

Install SynHoPy from PyPI:

# Core verifier
pip install synhopy

# Optional but strongly recommended — enables Z3-backed proofs
pip install z3-solver

Without z3-solver SynHoPy can still run in LLM-judged mode, but you will not get the strongest (Z3_PROVEN) trust level.

Confirm the installation:

synhopy --version
# synhopy 0.4.0 (z3 available: yes)

If you see z3 available: yes you are ready to go. If it says no, double-check that z3-solver is installed in the same environment.

Your First Verified Function

Create a file called mymath.py with the following contents. This is plain Python — the only addition is the @guarantee decorator that states what the function promises to its callers. 

from __future__ import annotations
from synhopy import guarantee


@guarantee("result >= a and result >= b and (result == a or result == b)")
def max_of(a: int, b: int) -> int:
    """Return the larger of two integers."""
    if a >= b:
        return a
    else:
        return b

Notice that the function body is completely ordinary Python. There are no proof terms, no monadic wrappers, and no special base classes. SynHoPy reads the AST, extracts proof obligations from the @guarantee specification, and discharges them automatically.

Running the Verifier

Verify the file from your terminal:

synhopy verify mymath.py

You should see output like this:

──────────────────────────────────────────────────
 SynHoPy v0.4.0 — verifying mymath.py
──────────────────────────────────────────────────

max_of(a: int, b: int) -> int
  Proof obligations:
    1. result >= a            ✅  Z3_PROVEN
    2. result >= b            ✅  Z3_PROVEN
    3. result == a or result == b  ✅  Z3_PROVEN

  Status: ✅ VERIFIED   Trust: Z3_PROVEN   Time: 0.03s

──────────────────────────────────────────────────
 1 function verified · 0 failed · 3 obligations discharged
──────────────────────────────────────────────────
Continuous verification. Run synhopy watch mymath.py and the verifier will re-check the file every time you save. This pairs well with a split terminal — code on the left, verification output on the right.

Two Styles of Specification: @guarantee vs @ensures

SynHoPy provides two complementary ways to state postconditions, designed around the principle of gradual formality:

@guarantee — Natural Language

from synhopy import guarantee

@guarantee("result is non-negative")
def square(n: int) -> int:
    return n * n
  • Takes a plain English string
  • SynHoPy parses it into Z3 assertions when possible, falls back to the LLM oracle otherwise
  • Yields Z3_PROVEN when the string maps to a decidable formula, or LLM_JUDGED when it cannot
  • Best for: quick annotations, documentation-grade specs, early-stage verification

@ensures — Formal Predicate

from synhopy import ensures

@ensures(lambda n, result: result >= 0)
def square(n: int) -> int:
    return n * n
  • Takes a Python callable (args..., result) → bool
  • SynHoPy encodes it directly into Z3 — no NL parsing, no oracle
  • Always yields Z3_PROVEN or a concrete counterexample
  • Best for: machine-checkable contracts, CI gates, production-grade proofs

Similarly, preconditions have two forms: 

@requires("xs is non-empty")  # NL string
@requires(lambda xs: len(xs) > 0)  # formal
The gradual formality principle. Start with @guarantee strings to capture intent quickly. As your project matures, tighten critical specs to @ensures lambdas for machine-checkable, Z3-proven guarantees. Both can coexist on the same function: 
@guarantee("returns the sorted, deduplicated version of xs")
@ensures(lambda xs, r: len(r) <= len(xs))
@ensures(lambda xs, r: all(r[i] < r[i+1] for i in range(len(r)-1)))
def unique_sorted(xs: list[int]) -> list[int]:
    return sorted(set(xs))
Here the two @ensures predicates are Z3-proven; the @guarantee string acts as human-readable documentation and may trigger an additional oracle check for the "deduplication" semantics that Z3 cannot express directly.

Understanding the Output

Each function annotated with @guarantee produces a verification block in the output. Here is what each piece means:

ElementMeaning
Proof obligations Individual clauses extracted from the guarantee string. SynHoPy splits conjunctions (and) into separate obligations so you can see exactly which parts pass or fail.
Trust level How the obligation was discharged. From strongest to weakest:
🟢 LEAN_VERIFIED — machine-checked by Lean's kernel
🟡 Z3_PROVEN — proved by the Z3 SMT solver
🟠 LLM_JUDGED — an LLM oracle judged it correct (confidence ≥ threshold)
🔴 UNTRUSTED — no verification yet
Status ✅ VERIFIED if every obligation is discharged at the configured minimum trust level; ❌ FAILED otherwise.
Time Wall-clock time for that function's verification.

A Slightly Harder Example

Let's verify an absolute-value function. The guarantee now involves conditionals that SynHoPy must reason about across both branches. 

from __future__ import annotations
from synhopy import guarantee


@guarantee("result >= 0 and (result == x or result == -x)")
def abs_val(x: int) -> int:
    """Return the absolute value of x."""
    if x >= 0:
        return x
    else:
        return -x

Running synhopy verify on this file produces:

abs_val(x: int) -> int
  Proof obligations:
    1. result >= 0                       ✅  Z3_PROVEN
    2. result == x or result == -x      ✅  Z3_PROVEN

  Status: ✅ VERIFIED   Trust: Z3_PROVEN   Time: 0.02s

Z3 handles both branches by case-splitting on the condition x >= 0. In each branch it checks the guarantee independently and merges the results.

When Verification Fails

What happens if the code is wrong? Suppose we make a mistake in our max_of implementation: 

from __future__ import annotations
from synhopy import guarantee


@guarantee("result >= a and result >= b and (result == a or result == b)")
def max_of(a: int, b: int) -> int:
    """Oops — always returns the first argument."""
    return a  # BUG: ignores b entirely

The verifier will catch this immediately:

max_of(a: int, b: int) -> int
  Proof obligations:
    1. result >= a            ✅  Z3_PROVEN
    2. result >= b            ❌  FAILED
    3. result == a or result == b  ✅  Z3_PROVEN

  Status: ❌ FAILED

  Counterexample (from Z3):
    a = 0, b = 1
    result = 0
    Violated: result >= b  (0 >= 1 is False)

──────────────────────────────────────────────────
 0 functions verified · 1 failed · 2/3 obligations discharged
──────────────────────────────────────────────────

The counterexample is concrete: when a = 0 and b = 1, the function returns 0, which violates result >= b. This is one of the great strengths of SMT-backed verification — when something is wrong, you get a witness, not just a vague error.

Mixing verified and unverified code. A @guarantee only tells you that the decorated function satisfies its postcondition assuming it is called with values that satisfy Python's type annotations. If unverified code passes bad data (e.g., a string where an int is expected), the guarantee says nothing about the outcome. Keep verification boundaries clean: verify entire modules, not isolated functions surrounded by untyped code.

IDE Integration

SynHoPy ships with a VS Code extension (synhopy-vscode) that shows verification results inline as you type. Verified guarantees are marked with a green gutter icon; failures show red with the counterexample on hover. 

# In VS Code with synhopy-vscode installed, you see:
#
#  ✅ │ @guarantee("result >= a and result >= b ...")
#     │ def max_of(a: int, b: int) -> int:
#     │     if a >= b:
#     │         return a
#     │     else:
#     │         return b
#
# The green checkmark appears once verification completes.
# Hover it for details (trust level, time, obligations).

Install it from the VS Code marketplace or via the command line: 

code --install-extension synhopy.synhopy-vscode

Project Configuration

For multi-file projects, place a synhopy.toml at your project root. Here is a minimal example: 

# synhopy.toml
[synhopy]
python   = "3.12"
min_trust = "Z3_PROVEN"     # reject LLM-only judgements
z3_timeout = 30            # seconds per obligation

[synhopy.paths]
include = ["src/**/*.py"]
exclude = ["tests/**"]

With a config file in place you can simply run synhopy verify (no filename) and it will verify every file matched by include.

What SynHoPy Checks (and What It Doesn't)

At this stage SynHoPy is checking functional correctness — does the return value satisfy the postcondition for every possible input that matches the type annotations? It does not yet check:

  • Termination (covered in Chapter 6)
  • Side effects or mutation (covered in Part 6)
  • Concurrency safety (also Part 6)

Those topics build on the foundation laid here. For now, focus on pure functions with clear input/output contracts.

A Complete Module

Let's put several verified functions together into a small utility module to see how SynHoPy handles a realistic file. 

from __future__ import annotations
from synhopy import guarantee, assume


@guarantee("result >= 0")
def square(n: int) -> int:
    return n * n


@guarantee("result >= lo and result <= hi")
def clamp(x: int, lo: int, hi: int) -> int:
    assume("lo <= hi")
    if x < lo:
        return lo
    elif x > hi:
        return hi
    else:
        return x


@guarantee("result == a + b")
def add(a: int, b: int) -> int:
    return a + b


@guarantee("(result == True and n > 0) or (result == False and n <= 0)")
def is_positive(n: int) -> bool:
    return n > 0

Running synhopy verify utils.py:

──────────────────────────────────────────────────
 SynHoPy v0.4.0 — verifying utils.py
──────────────────────────────────────────────────

square(n: int) -> int
  1. result >= 0   ✅  Z3_PROVEN
  Status: ✅ VERIFIED   Trust: Z3_PROVEN   Time: 0.01s

clamp(x: int, lo: int, hi: int) -> int
  Precondition: lo <= hi
  1. result >= lo   ✅  Z3_PROVEN
  2. result <= hi   ✅  Z3_PROVEN
  Status: ✅ VERIFIED   Trust: Z3_PROVEN   Time: 0.02s

add(a: int, b: int) -> int
  1. result == a + b   ✅  Z3_PROVEN
  Status: ✅ VERIFIED   Trust: Z3_PROVEN   Time: 0.01s

is_positive(n: int) -> bool
  1. (result == True and n > 0) or
     (result == False and n <= 0)   ✅  Z3_PROVEN
  Status: ✅ VERIFIED   Trust: Z3_PROVEN   Time: 0.01s

──────────────────────────────────────────────────
 4 functions verified · 0 failed · 5 obligations discharged
──────────────────────────────────────────────────

Note how clamp uses assume("lo <= hi") to declare a precondition. SynHoPy adds this as an axiom when proving the postcondition, and separately generates a proof obligation at every call site of clamp to ensure the precondition actually holds.

Exercise

Exercise 1 — clamp with guarantee.

Write a function clamp(x: int, lo: int, hi: int) -> int that returns lo if x < lo, hi if x > hi, and x otherwise. Add a @guarantee that the result is between lo and hi (inclusive), and an assume("lo <= hi") precondition.

Verify your solution with synhopy verify. Then deliberately introduce a bug (e.g., swap the lo and hi returns) and observe the counterexample that Z3 produces.

Hint

The guarantee string should be "result >= lo and result <= hi". Don't forget from __future__ import annotations.
