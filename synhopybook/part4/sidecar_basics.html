Sidecar Proof Basics

This chapter teaches you how to create sidecar specifications — formal contracts for libraries you don't own. By the end, you'll be able to write .synhopy files for any Python module and verify your code against them.

Creating a .synhopy File

A sidecar file mirrors the structure of the module it describes. For a module foo.bar, create foo/bar.synhopy in your project's sidecars/ directory (or wherever you configure synhopy.toml to look). 

# Project layout
myproject/
├── src/
│   └── my_app.py          # your code — imports math, collections
├── sidecars/
│   ├── math.synhopy        # sidecar for Python's math module
│   └── collections.synhopy # sidecar for Python's collections
├── synhopy.toml            # configuration
└── tests/

The synhopy.toml configuration:

# synhopy.toml
[sidecar]
search_paths = ["sidecars/"]
auto_discover = true

[verification]
default_trust = "Z3_PROVEN"
timeout = 30  # seconds per proof obligation

The @about Decorator

Every specification in a sidecar file is bound to an external function or class via the @about decorator. The argument is the fully qualified name of the target. 

from synhopy.sidecar import about, ExternalSpec

@about("math.sqrt")
class SqrtSpec(ExternalSpec):
    """Specification for math.sqrt."""
    ...

@about("math.factorial")
class FactorialSpec(ExternalSpec):
    """Specification for math.factorial."""
    ...

The name resolution rules are:

  • "math.sqrt" — top-level function in a module
  • "collections.OrderedDict" — a class
  • "collections.OrderedDict.popitem" — a method on a class
  • "numpy.linalg.solve" — nested module function
SynHoPy verifies that the target exists in the installed package at verification time. If math.sqrt doesn't resolve, you get a clear error — not a silent failure.

The ExternalSpec Class

ExternalSpec is the base class for all sidecar specifications. It provides a structured way to declare preconditions, postconditions, algebraic laws, and exception behavior. 

from synhopy.sidecar import about, ExternalSpec, assume, prove, law

@about("math.sqrt")
class SqrtSpec(ExternalSpec):
    """math.sqrt(x) → float"""

    def precondition(self, x: float) -> bool:
        """x must be non-negative."""
        return x >= 0

    def postcondition(self, x: float, result: float) -> bool:
        """Result is the non-negative square root."""
        return result >= 0 and abs(result * result - x) < 1e-10

    def raises(self, x: float) -> type | None:
        """ValueError when x < 0."""
        if x < 0:
            return ValueError
        return None

    @law
    def sqrt_of_square(self, x: float) -> bool:
        """sqrt(x²) == |x| for all x."""
        return abs(self.call(x * x) - abs(x)) < 1e-10

ExternalSpec Methods

Method Purpose Required?
precondition(self, ...) Conditions that must hold before calling the function Optional
postcondition(self, ..., result) Conditions guaranteed to hold after the call returns Optional
raises(self, ...) Which exception (if any) is thrown for given inputs Optional
@law methods Algebraic properties (commutativity, etc.) Optional
trust_level attribute How much to trust this spec (default: Z3_PROVEN) Optional

Assume vs. Prove

Sidecar specs distinguish between things you assume about the library and things you prove about your usage of it.

assume

# You TRUST the library does this
@about("math.sqrt")
class SqrtSpec(ExternalSpec):
    trust_level = "ASSUMED"

    @assume
    def postcondition(self, x, r):
        return r >= 0
# ↑ Not verified — taken on faith

prove

# SynHoPy CHECKS this at call sites
@about("math.sqrt")
class SqrtSpec(ExternalSpec):
    trust_level = "Z3_PROVEN"

    @prove
    def precondition(self, x):
        return x >= 0
# ↑ Z3 must prove this at every call

The rule is simple: postconditions are assumed (you trust the library's implementation), preconditions are proved (you must show your code satisfies them).

If you mark a postcondition with @prove, SynHoPy will try to verify the library's implementation — which requires source access. This only works for libraries you also have the source code for. For pip packages, stick with @assume on postconditions.

Example: Sidecar for Python's math Module

Let's build a complete sidecar for the most commonly used functions in math. 

# sidecars/math.synhopy
from synhopy.sidecar import about, ExternalSpec, assume, prove, law
from synhopy.types import Nat, PosInt, NonNegFloat

# ── sqrt ──────────────────────────────────────────────
@about("math.sqrt")
class SqrtSpec(ExternalSpec):
    def precondition(self, x: float) -> bool:
        return x >= 0

    @assume
    def postcondition(self, x: float, result: float) -> bool:
        return result >= 0 and abs(result * result - x) < 1e-10

    @law
    def monotone(self, a: NonNegFloat, b: NonNegFloat) -> bool:
        """sqrt is monotonically increasing."""
        if a <= b:
            return self.call(a) <= self.call(b)
        return True

# ── factorial ─────────────────────────────────────────
@about("math.factorial")
class FactorialSpec(ExternalSpec):
    def precondition(self, n: int) -> bool:
        return n >= 0

    @assume
    def postcondition(self, n: int, result: int) -> bool:
        return result > 0  # factorial is always positive

    @law
    def recursive_def(self, n: PosInt) -> bool:
        """n! == n * (n-1)!"""
        return self.call(n) == n * self.call(n - 1)

    @law
    def base_case(self) -> bool:
        """0! == 1"""
        return self.call(0) == 1

    def raises(self, n: int) -> type | None:
        if n < 0:
            return ValueError
        return None

# ── gcd ───────────────────────────────────────────────
@about("math.gcd")
class GcdSpec(ExternalSpec):
    @assume
    def postcondition(self, a: int, b: int, result: int) -> bool:
        """result divides both a and b."""
        if result == 0:
            return a == 0 and b == 0
        return (a % result == 0) and (b % result == 0) and result >= 0

    @law
    def commutative(self, a: int, b: int) -> bool:
        """gcd(a, b) == gcd(b, a)"""
        return self.call(a, b) == self.call(b, a)

    @law
    def idempotent(self, a: int) -> bool:
        """gcd(a, a) == |a|"""
        return self.call(a, a) == abs(a)

    @law
    def identity(self, a: int) -> bool:
        """gcd(a, 0) == |a|"""
        return self.call(a, 0) == abs(a)

Example: Sidecar for collections.OrderedDict

# sidecars/collections.synhopy
from synhopy.sidecar import about, ExternalSpec, assume, law

@about("collections.OrderedDict.__setitem__")
class SetItemSpec(ExternalSpec):
    @assume
    def postcondition(self, key, value, *, _self_before, _self_after):
        """Newly inserted key appears at the end of iteration order."""
        if key not in _self_before:
            return list(_self_after.keys())[-1] == key
        return True  # existing key retains its position

@about("collections.OrderedDict.popitem")
class PopItemSpec(ExternalSpec):
    def precondition(self, *, last: bool = True, _self) -> bool:
        """Dict must not be empty."""
        return len(_self) > 0

    @assume
    def postcondition(self, result: tuple, *, last=True, _self_before) -> bool:
        """When last=True, returns the last-inserted pair."""
        keys = list(_self_before.keys())
        expected_key = keys[-1] if last else keys[0]
        return result[0] == expected_key

    def raises(self, *, last=True, _self) -> type | None:
        if len(_self) == 0:
            return KeyError
        return None

@about("collections.OrderedDict.move_to_end")
class MoveToEndSpec(ExternalSpec):
    def precondition(self, key, *, last=True, _self) -> bool:
        return key in _self

    @assume
    def postcondition(self, key, *, last=True, _self_after) -> bool:
        keys = list(_self_after.keys())
        if last:
            return keys[-1] == key
        return keys[0] == key

Loading Sidecars: SidecarModule

SynHoPy provides SidecarModule to programmatically load and query sidecar specifications. 

from synhopy.sidecar import SidecarModule

# Load a sidecar
math_sidecar = SidecarModule.load("sidecars/math.synhopy")

# Query specs
sqrt_spec = math_sidecar.get_spec("math.sqrt")
print(sqrt_spec.precondition)   # x >= 0
print(sqrt_spec.postcondition)  # result >= 0 ∧ result² ≈ x
print(sqrt_spec.laws)           # [monotone, sqrt_of_square]
print(sqrt_spec.trust_level)    # TrustLevel.Z3_PROVEN

# List all specs in this sidecar
for name, spec in math_sidecar.specs.items():
    print(f"  {name}: {len(spec.laws)} laws, trust={spec.trust_level}")

Handling Exceptions in Specs

The raises method lets you specify which exceptions a function throws for given inputs. SynHoPy uses this in two ways:

  1. Negative preconditions: If you call math.sqrt(-1), SynHoPy knows a ValueError will be raised and flags it.
  2. Exception-safe code: If your code wraps calls in try/except, SynHoPy verifies you catch the right exception type.
# Your code with try/except
def safe_sqrt(x: float) -> float | None:
    try:
        return math.sqrt(x)
    except ValueError:
        return None

# SynHoPy verifies:
# 1. math.sqrt CAN raise ValueError (from the sidecar spec)
# 2. The except clause catches the right type
# 3. Return type is float | None (union of both branches)

Module Signatures: .synhopy-sig Files

For large libraries, writing full ExternalSpec classes for every function is tedious. SynHoPy supports a lightweight signature format for quickly declaring types without full pre/post conditions: 

# math.synhopy-sig — lightweight type signatures
# Format: function_name : (param_types) -> return_type [trust_level]

sqrt : (x: float {x >= 0}) -> float {result >= 0}  [Z3_PROVEN]
factorial : (n: int {n >= 0}) -> int {result > 0}  [Z3_PROVEN]
gcd : (a: int, b: int) -> int {result >= 0}  [Z3_PROVEN]
ceil : (x: float) -> int {result >= x}  [Z3_PROVEN]
floor : (x: float) -> int {result <= x}  [Z3_PROVEN]
log : (x: float {x > 0}) -> float  [Z3_PROVEN]
log2 : (x: float {x > 0}) -> float  [Z3_PROVEN]
exp : (x: float) -> float {result > 0}  [Z3_PROVEN]
sin : (x: float) -> float {-1 <= result <= 1}  [Z3_PROVEN]
cos : (x: float) -> float {-1 <= result <= 1}  [Z3_PROVEN]

Signature files are automatically combined with full .synhopy specs. If both exist, the .synhopy file takes precedence.

JSON Spec Format for CI

For CI integration, SynHoPy can export and consume specifications in JSON: 

$ synhopy sidecar export sidecars/math.synhopy --format json

{
  "module": "math",
  "version": "3.12",
  "specs": {
    "math.sqrt": {
      "precondition": "x >= 0",
      "postcondition": "result >= 0 and abs(result**2 - x) < 1e-10",
      "raises": { "ValueError": "x < 0" },
      "laws": [
        { "name": "monotone", "formula": "a <= b => sqrt(a) <= sqrt(b)" },
        { "name": "sqrt_of_square", "formula": "sqrt(x**2) == abs(x)" }
      ],
      "trust_level": "Z3_PROVEN"
    },
    "math.factorial": {
      "precondition": "n >= 0",
      "postcondition": "result > 0",
      "raises": { "ValueError": "n < 0" },
      "laws": [
        { "name": "recursive_def", "formula": "f(n) == n * f(n-1)" },
        { "name": "base_case", "formula": "f(0) == 1" }
      ],
      "trust_level": "Z3_PROVEN"
    }
  }
}

Use the JSON format in CI pipelines to track specification drift across versions: 

# .github/workflows/verify.yml (excerpt)
- name: Check sidecar specs
  run: |
    synhopy sidecar verify src/ --sidecar-dir sidecars/ --report json > report.json
    synhopy sidecar diff report.json baseline.json --fail-on-regression

Best Practices

1. Start small. Don't try to specify every function in NumPy. Start with the functions your safety-critical code actually calls.
2. Specify only what you rely on. If your code only needs sqrt(x) >= 0, don't also specify floating-point precision — keep specs minimal and maintainable.
3. Use algebraic laws for structural properties. Laws like commutativity and associativity compose well and catch subtle bugs.
4. Version-pin your sidecars. Library updates can break specs. Include the library version in your sidecar metadata so CI catches API drift.
