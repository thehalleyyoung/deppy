Overview

Aims

The core aims of deppy: zero-change verification, type-driven generation, anti-hallucination, and a mathematically honest story for LLM-assisted software.

On this page

deppy tries to close several gaps at once: the gap between code and docs, the gap between natural-language intent and executable software, the gap between local checks and global correctness, and the gap between impressive demos and tools that people actually use.

Primary aims

  • Zero-change verification. Point deppy at an existing codebase and extract implicit specifications from code, docstrings, names, asserts, tests, and structure.
  • Type-driven generation. Expand a natural-language spec into a typed project plan, then into code, tests, proof obligations, and integration checks.
  • Controlled use of LLMs. LLM outputs live inside an oracle discipline with provenance, confidence, budgets, and weakest-link trust propagation.
  • Anti-hallucination by construction. Hallucination is a failure to inhabit a required hybrid type.
  • Verified planning. A plan is an expanded dependent type whose overlaps must satisfy a sheaf condition.
  • Interdisciplinary ideation. Creative cross-domain insight becomes a typed gluing problem over a cover of domains.

The philosophical claim

The central claim is that software verification becomes far more usable when it stops pretending all truth lives in one layer. A real project has intent, type structure, semantic tests, proof artifacts, and implementation details.

\[\mathrm{HybridTy} : (\mathrm{Site}(P) \times \mathrm{Layer})^{op} \to \mathrm{RefinementLattice} \times \mathrm{TrustLattice}.\]

Why a website like this exists

deppy needs multiple entry points: some readers want a command, some want an API, some want a theorem statement, and some want the categorical picture. The docs are split accordingly.
