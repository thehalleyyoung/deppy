Two layers of typing, one repository

deppy is easiest to misread when the reader thinks the plain type system and the hybrid type system are rival explanations. They are not. The plain system is the foundational language of types, refinements, binders, witnesses, and subtype obligations. The hybrid system lifts that language into a five-layer object that can carry intent, solver-backed structure, oracle-judged semantics, proofs, and executable code at once. The rest of the repo mostly makes sense as a consequence of that lift.

\[\text{plain types} \subset \text{the language of obligations},\qquad \text{hybrid types} = \text{plain content} + \text{layered evidence}.\]

Orientation

Fast answer: what plain and hybrid mean here

In the repository's own source code, the plain system is the foundational type hierarchy under `src/deppy/types/`. It supplies the semantic carriers, refinement types, dependent functions and pairs, equality-like objects, subtype relations, indexed families, and the support machinery for variables and witnesses. If you need a place where types are represented, compared, traversed, substituted, or reduced to obligations, the plain layer is the answer.

The hybrid system lives under `src/deppy/hybrid/` and adds a five-point verification layer structure, trust levels, provenance tracking, oracle-mediated semantic predicates, Lean translation, staged discharge, anti-hallucination, type expansion, zero-change checking, and a prompt-driven generation story. It keeps depending on the plain layer for the shape of many obligations, but it refuses to collapse all evidence into one undifferentiated “verified / not verified” bit.

So the shortest precise sentence is this: the plain system tells deppy what a type-theoretic object is, while the hybrid system tells deppy how that object is represented and checked across intent, structure, semantics, proof, and code.

Plain core

The foundational plain system in detail

The base docstring of `src/deppy/types/base.py` describes the module as the “base type hierarchy for DepPy's sheaf-descent semantic typing system”. That wording matters because it means the foundational layer is already geometric and semantic in self-conception. It is not a purely nominal shadow of Python annotations. The file presents types as carriers for local sections in a sheaf-descent framework, and the rest of `deppy.types` elaborates that language into a genuine dependent-and-refinement toolkit.

Every concrete plain type inherits from `TypeBase`, which requires substitution, free-variable enumeration, traversal, equality, hashing, and representation. Those operations are the essential algebra of a symbolic type language. Because they are available everywhere, later modules can build subtyping, theorem translation, witness extraction, render pipelines, and hybrid lifting without inventing separate ad hoc representations.

The foundational plain system also gives the repository a vocabulary rich enough to discuss ordinary Python programs and not just theorem examples. Primitive types, top and bottom, containers, callables, class-like forms, modules, protocols, optionals, unions, tuples, sets, and mapping-oriented structures all live here. In other words, plain DepPy is intended to be an expressive semantic typing framework for Python and Python-adjacent reasoning, not merely a decorative front-end for the hybrid story.

Plain system

src/deppy/types/base.py

The foundational module describes DepPy's base type hierarchy as the carrier language for sheaf-descent semantic typing. `TypeBase` requires substitution, free-variable computation, structural traversal, equality, hashing, and readable representations. The file explicitly frames types as carriers for local sections in a sheaf-descent framework, so the language of plain types is already geometric rather than merely nominal.

\[\mathrm{TypeBase} \ni \tau \mapsto (\mathrm{substitute},\ \mathrm{free\_variables},\ \mathrm{walk})\]\n\[\mathrm{Any} \text{ is top},\qquad \mathrm{Never} \text{ is bottom}.\]

The plain hierarchy starts with primitives, top and bottom (`AnyType`, `NeverType`), then moves through container forms such as `ListType`, `DictType`, `TupleType`, `SetType`, and callable or class-oriented forms. This makes the plain system feel like a typed semantic model of Python rather than a theorem-proving toy language detached from actual programs.

Substitution is a first-class operation because every later dependent construction depends on hygienic replacement of binders and free variables. In practice that means the plain system is built to support symbolic manipulation, normalization, and comparison, which is exactly what later subtyping, theorem extraction, and translation layers need.

Because `TypeBase.walk()` recursively traverses a type expression, many later passes treat the plain hierarchy as an inspectable syntax tree. This is how the plain type system quietly becomes infrastructure for renderers, refinement explainers, Lean emitters, and bridge layers that need to map types into other semantic domains.

Plain system

src/deppy/types/refinement.py

The plain refinement layer defines the canonical shape `\{v : au \mid arphi\}`. `Predicate` is an abstract interface with free-variable analysis, substitution, negation, equality, hashing, and combination operators. Concrete predicates cover truth, falsity, comparison, conjunction, disjunction, implication, membership, and other light-weight forms suitable for static and solver-backed reasoning.

\[\{v : \tau \mid \varphi(v)\}\]\n\[\varphi = \varphi_1 \wedge \varphi_2 \wedge \cdots \wedge \varphi_n\]

The refinement module is the clearest statement of what the plain system thinks a specification is: a base type together with a decidable or at least programmatically manipulable predicate. Qualifiers such as positivity, non-nullness, non-emptiness, and sortedness give the system a reusable vocabulary of named refinements.

What matters architecturally is that this refinement language is intentionally static. There is no trust score, no provenance, and no confidence in the plain `RefinementType`. It is an object in a semantic lattice, not an evidence-carrying value. That distinction is exactly why the hybrid system later has to split predicates into structural and semantic families.

Even so, the plain refinement layer remains central after hybridization. Mixed-mode surface syntax compiles many user-facing promises into plain refinement obligations, and Lean translation continues to treat refinement formulas as formal objects rather than as marketing text. So the plain system is not obsolete; it is the hard logical substrate on which the new semantic machinery sits.

Plain system

src/deppy/types/dependent.py

The plain dependent module provides `PiType`, `SigmaType`, `IdentityType`, `IndexedFamily`, `WitnessType`, `ForallType`, and `ExistsType`. `PiType` models dependent functions `\Pi(x : au_1). au_2`, while `SigmaType` models dependent pairs `\Sigma(x : au_1). au_2`. The code includes helper operations like `apply`, `alpha_rename`, `is_dependent`, and second-projection specialization.

\[\Pi(x : A). B(x)\qquad\qquad \Sigma(x : A). B(x)\]\n\[\mathrm{Id}_A(a,b)\]

This module is where plain DepPy becomes recognizably dependent type theory rather than just enriched Python typing. The binder-sensitive behavior of `PiType` and `SigmaType` gives the system a language for APIs whose outputs depend on inputs and for witness-bearing values whose proofs or subordinate objects depend on a first component.

The existence of `IdentityType` in the plain system matters because it prevents the hybrid story from pretending equality appeared only after the LLM arrived. Compatibility on overlaps, path-style identifications, and bridge conditions already have a home in the foundational system; hybridization mostly broadens the set of layers across which those equalities have to be respected.

When the mixed-mode compiler maps `hole()` toward Sigma-style obligations and `spec()` toward refinement obligations, it is directly exploiting the plain dependent language. That is why the right way to read the hybrid stack is extension rather than replacement: plain Pi and Sigma continue to bear the mathematical weight.

Plain system

src/deppy/types/subtyping.py

The plain system includes a dedicated subtype checker that can answer subtype questions and emit verification conditions. This module is where refinement and dependent structure become proof obligations rather than only syntax. In other words, it is the part of the plain system that operationalizes the refinement lattice.

\[\tau_1 \leq \tau_2\]\n\[\mathrm{VC}(\tau_1,\tau_2) = \text{proof obligations needed for } \tau_1 \subseteq \tau_2\]

Subtyping is not just an implementation detail. Once a type system offers top and bottom, refinements, callable structure, tuples, dictionaries, and dependent variants, the subtype relation becomes the machine that orders the semantic universe. Many later hybrid checks still rely on the fact that the plain core knows how to compare and discharge relationships among type expressions.

By representing verification conditions explicitly, the plain system offers an abstraction boundary for later Z3 and proof tooling. The hybrid system can layer trust and semantic judgments on top, but the actual obligation structure owes a great deal to this earlier, more classical checker architecture.

In descriptive terms, the plain type system says what should hold and how symbolic type expressions relate; the hybrid system says how strongly we trust each piece of evidence that those relationships are satisfied in real generated or existing code.

Plain system

src/deppy/types/indexed.py, carriers.py, witnesses.py, variables.py

The remaining plain modules provide the smaller but essential parts of the semantic toolkit: indexed families for parameterized types, carrier and witness representations, and variable machinery for substitution and scope. They are easy to underrate because they are less glamorous than Pi or cohomology, but they are exactly what lets the foundational system behave like a manipulable formal language rather than a set of ad hoc Python classes.

\[\mathcal{F}(i)\ \text{for an indexed family}\qquad\qquad \exists w.\ \mathrm{Witness}(w)\]

Indexed families matter because many real mathematical and programmatic objects are not stand-alone types but parameterized ones: vectors by length, tensors by shape, records by schema, and APIs by configuration. Without indexed families the rest of the stack would have to fake dependency through naming conventions.

Witness and carrier machinery give the plain system a place to store evidence and payloads together when necessary, especially in theorem-oriented or contract-oriented contexts. That becomes important again in hybrid mode, where witness-like objects acquire trust metadata and provenance chains.

The variable subsystem keeps the entire plain hierarchy honest. A dependent type system without reliable variable bookkeeping is a pile of notation. DepPy's plain system therefore spends real engineering effort on free variables, renaming, and substitution, and the hybrid layer inherits those semantics rather than inventing its own ad hoc binder rules.

Plain significance

What the plain system is really buying the repo

The plain system gives the repository exact symbolic content. When a refinement says “positive”, the plain layer decides what that means as a formal predicate object rather than leaving the phrase suspended in prose. When a function is dependent, the plain layer knows whether the binder actually appears in the codomain and how to substitute an argument into the result type. When a subtype relation is needed, the plain layer determines how that relation is expressed and discharged.

That symbolic content is what later makes a mixed-mode compilation story possible. Surface syntax such as `@guarantee`, `@spec`, and `hole()` only becomes meaningful because the compiler can lower it into plain dependent and refinement obligations. If the plain system did not exist, the surface language would have nowhere mathematically respectable to land.

The plain layer also stabilizes the project's research ambitions. Cohomology, Lean translation, and trust lattices may sound like the exciting parts, but they still need an object language to speak about. The plain types are that object language. They give the rest of deppy something exact to transport, compare, restrict, prove, or fail to glue.

Hybrid core

The hybrid system in detail

The docstring of `src/deppy/hybrid/core/types.py` names the core object unambiguously: a hybrid type is a global section of `HybridTy : (Site(P) × Layer)^{op} → RefinementLattice × TrustLattice`. The file immediately names the five-layer poset `INTENT < STRUCTURAL < SEMANTIC < PROOF < CODE`, the requirement that first cohomology vanish for coherence, and the oracle monad `T_O(A) = A × TrustLevel × Confidence × Provenance` as the semantic wrapper for oracle judgments.

Those choices tell you what hybridization means. It does not simply mean “use an LLM somewhere”. It means taking the foundational type-theoretic content of the project and interpreting it over a product site where different kinds of evidence and artifacts can coexist. Intent is not treated as disposable prose. Structural facts are not conflated with semantic judgments. Proofs are not collapsed into solver results. Code is not taken as self-authenticating merely because it runs. All of those layers become coordinates of one typed object.

The hybrid system therefore adds epistemic structure, pipeline structure, and gluing structure. Epistemic structure is trust, confidence, provenance, and evidence. Pipeline structure is the ordered movement from intent through structure and semantics toward proof and code. Gluing structure is the insistence that local sections and adjacent layers agree or else exhibit an obstruction.

Hybrid system

src/deppy/hybrid/core/types.py

The central hybrid module states the project's main mathematical slogan explicitly: a `HybridType` is a global section of `HybridTy : (Site(P) imes Layer)^{op} o RefinementLattice imes TrustLattice`. The layer poset is `INTENT < STRUCTURAL < SEMANTIC < PROOF < CODE`, and coherence is expressed by the vanishing of first cohomology. This file therefore plays three roles at once: it defines concrete Python data structures, it explains their mathematical interpretation, and it exposes the public API that the rest of the hybrid stack consumes.

\[\mathrm{HybridTy} : (\mathrm{Site}(P) \times \mathrm{Layer})^{op} \to \mathrm{RefinementLattice} \times \mathrm{TrustLattice}\]\n\[H^1(\mathrm{Layer},\mathrm{HybridTy}) = 0\]

The five sections are not just labels. `IntentSection` stores natural-language specifications and structured claims. `StructuralSection` stores conjunctions of machine-checkable predicates, often backed by Z3 or Python evaluation. `SemanticSection` stores oracle-judged predicates with rubrics, prompts, and confidence thresholds. `ProofSection` stores Lean-facing theorem material, and `CodeSection` stores executable code and structural program representations.

A key conceptual move here is that trust is not a side table. It is built into the type-theoretic object through the trust lattice and through the oracle monad. That means a hybrid type does not merely assert that a claim exists; it carries how the claim was checked, how strong that evidence is, and what provenance chain produced it. The system is explicitly refusing to erase epistemology.

The same file also defines the hybrid versions of familiar type formers: `HybridPiType`, `HybridSigmaType`, and `HybridRefinementType`. Those are the precise sense in which the hybrid world extends the plain one. The familiar dependent syntax remains, but now each type former can reference structural and semantic predicates, layer restrictions, and trust propagation.

Hybrid system

src/deppy/hybrid/core/types.py (Layer, TrustLevel, Z3 helpers)

The hybrid core duplicates some names intentionally so imports remain cheap and circularity stays manageable. It defines `Layer`, the ordering map `LAYER_ORDER`, a trust lattice with meet and join, plus Z3-oriented helper structures such as `Z3Sort` and `Z3Type`. This is one of the places where the page must be precise: parts of the hybrid package export lightweight convenience enums in `deppy.hybrid.__init__`, while the deeper core defines the full trust machinery used by the main checker.

\[\text{INTENT} < \text{STRUCTURAL} < \text{SEMANTIC} < \text{PROOF} < \text{CODE}\]\n\[a \sqcap b = \min(a,b),\qquad a \sqcup b = \max(a,b)\]

The trust lattice is not merely a ranking; it is algebraic infrastructure. `meet` is the weakest-link operation used when composing evidence, while `join` is the strongest-evidence operation used when aggregating competing proof sources. This is a crucial design choice because it makes the system compositional rather than slogan-based.

Structural predicates live in the hybrid setting as explicit objects rather than as plain booleans. They can be checked through Z3 or Python evaluation, conjoined and disjoined, negated, and transported across layers. This makes the structural layer a rigorous descendant of the plain refinement layer rather than an unrelated runtime filter.

The result is that the hybrid system can talk about exact solver-backed facts and still keep them in the same overall typed object as semantic claims, Lean artifacts, and concrete code. The structural layer is therefore a hinge between classical decidable typing and the broader trust-aware world.

Hybrid system

src/deppy/hybrid/core/checker.py

The checker module is the operational heart of the hybrid story. It provides multiple checking modes such as structural-only, semantic-only, full, and Lean-oriented paths, and it includes semantic caching so unchanged content does not repeatedly consume oracle budget. This file is where the mathematical objects become an executable verification algorithm.

\[\Gamma \vdash e \Rightarrow T\qquad\qquad \Gamma \vdash e \Leftarrow T\]\n\[\mathrm{cache\ key} = (\mathrm{predicate},\ \mathrm{content\_hash})\]

The checker performs synthesis and checking in the familiar bidirectional style, but because it is hybrid, it can separate and then recombine the distinct proof obligations of structural, semantic, and proof layers. That gives deppy a disciplined answer to a hard question: how should solver-backed facts and LLM-judged facts coexist in one pipeline without pretending to be the same kind of evidence?

Its fixed-point flavor matters too. The checker does not merely validate one frozen artifact; it can iteratively refine its understanding, leveraging existing kernel and solver infrastructure. In this sense the hybrid checker is not just a validator but a mediator between the old static semantics and the new trust-aware pipeline.

Caching semantic checks by content hash is also conceptually important. It treats oracle judgments as expensive semantic resources with provenance, not as ephemeral side effects. That design ties the hybrid checker directly to production engineering concerns such as reproducibility, cost control, and auditability.

Hybrid system

src/deppy/hybrid/core/trust.py, provenance.py, intent_bridge.py

Several core modules fill in the infrastructure that lets hybrid types carry more than formulas. Trust and provenance objects record how evidence was obtained, while the intent bridge turns natural-language statements into structured claims that can feed later type expansion, semantic predicates, or contracts. These modules are why the hybrid system can treat the gap between a docstring and a theorem as typed territory rather than as an informal chasm.

\[T_O(A) = A \times \mathrm{TrustLevel} \times \mathrm{Confidence} \times \mathrm{Provenance}\]

In the plain system a type can say what should hold. In the hybrid system, provenance explains where the claim came from, trust classifies the strength of the evidence, and the intent bridge turns informal description into material that downstream checkers can consume. The result is a multi-stage semantics for specification itself.

That is precisely why deppy can talk about intent, structure, semantics, proof, and code as layers of one object. Each layer needs its own representation and its own restrictions, but they must still compose into a coherent story. Provenance and trust keep the story honest.

This is also the point where the repo occasionally overstates by sounding as though every semantic claim has a single canonical formalization. In practice the bridge is heuristic and lossy. The page should therefore present the intent layer as an input to formalization rather than as automatically complete formal content.

Hybrid system

src/deppy/hybrid/mixed_mode/syntax.py, parser.py, compiler.py

The mixed-mode surface layer is how the hybrid system becomes pleasant Python instead of a hidden internal engine. The syntax module promises a critical invariant: every surface form is valid Python before compilation. `hole()` raises `NotImplementedError` in uncompiled mode, `assume()` and `check()` return `True`, and decorators such as `@spec` or `@guarantee` pass through before compilation and verification.

\[\texttt{hole(\"desc\") } \leadsto \Sigma\text{-style obligation}\]\n\[\texttt{@spec(\"desc\") } \leadsto \text{refinement/spec obligation}\]

This graceful degradation is not cosmetic. It means editors, test runners, and standard Python tooling remain useful before a hybrid compilation pass has happened. The system is designed so users can write ordinary code with embedded natural-language fragments and then let the hybrid pipeline harvest, compile, verify, and discharge those fragments later.

The compiler side of mixed mode is where the relationship to the plain system becomes explicit again. Surface forms are translated into plain obligations such as `SigmaType` and `RefinementType`, then wrapped into hybrid contracts and trust-bearing workflows. That is why the right mental model is “surface syntax compiles down into the old type language, then back up into the hybrid pipeline”.

This surface layer also explains much of deppy's broader appeal: examples, tutorials, trading demos, and agent-generated artifacts all advertise readable decorators because the underlying hybrid language would otherwise be too deep for day-to-day use.

Hybrid system

src/deppy/hybrid/lean_translation/*, discharge/cascade.py

The hybrid system does not stop at saying which obligations exist; it includes a translation path into Lean 4 plus a staged discharge cascade. The cascade usually runs through trivial discharge, Z3, oracle-backed proof assistance, Lean checking, and residual fallback. This is one of the strongest examples of how deppy treats trust as graduated rather than binary.

\[\text{TRIVIAL} \to \text{Z3} \to \text{LLM\_PROVER} \to \text{LEAN\_CHECK} \to \text{RESIDUAL}\]

A subtle but important point is that semantic predicates cannot magically become theorem-prover facts. The Lean translation layer therefore emits axioms or assumptions for certain semantic claims, making the epistemic boundary explicit. That honesty is more important than pretending every oracle judgment is a theorem.

Because discharge results record the stage that solved an obligation, the output is inspectable and classifiable. A proof that passed through Lean after Z3 simplification carries different weight from one that remained residual or assumption-backed. Again, the hybrid system insists that provenance and evidence strength remain visible.

The discharge cascade is therefore both a proof engine and a trust refinery. It systematically tries stronger or more formal routes before settling for weaker ones, and its outputs can be summarized in reports, dashboards, or project manifests.

Hybrid system

src/deppy/hybrid/pipeline/*, type_expansion/*, anti_hallucination/*, bridge.py

The hybrid package fans out into pipeline orchestration, large-scale type expansion, anti-hallucination, and bridges back to existing deppy infrastructure. Type expansion grows a short natural-language spec into a typed plan; anti-hallucination checks treat fabricated APIs or semantically wrong code as type inhabitation failures; bridge modules connect presheaves, cohomology, solvers, and legacy contracts to the new layer stack.

\[\varphi = \varphi_d \land \varphi_s\]\n\[\text{hallucination} = \text{type inhabitation failure}\]

This is the part of the repository that most directly turns the hybrid type system into a system-wide operating model. The hybrid pipeline orders ideation, specification, synthesis, structural checking, semantic checking, Lean translation, discharge, and production. Every one of those stages can be read as manipulating a hybrid typed object rather than a bag of unrelated files.

Type expansion deserves special emphasis because it shows how hybrid types influence architecture, not just verification. A natural-language prompt is elaborated into module plans, contracts, interfaces, and eventually code-generation obligations. That means the type system participates in program design, not only in post-hoc checking.

Anti-hallucination is perhaps the most public-facing reuse of the theory. The repo repeatedly frames hallucination as type inhabitation failure. Structural hallucinations are fabricated APIs or impossible signatures; semantic hallucinations are outputs that fail the oracle-tracked meaning judgment. That reuse of the same split between decidable and semantic constraints is exactly the point of the hybrid layer.

Layer stack

Layer-by-layer reading of a hybrid type

INTENT

The intent layer is where natural-language specification enters. It includes docstrings, comments, structured claims, and bridged specification fragments. This layer is not automatically formal truth; it is typed intent, meant to be parsed, compared, restricted, and transported into more rigid layers.

STRUCTURAL

The structural layer houses solver-backed or otherwise decidable predicates. This includes shape constraints, length bounds, positivity, field presence, and similar facts that can be checked by Z3 or deterministic evaluation. Conceptually it is the hybrid descendant of the plain refinement layer.

SEMANTIC

The semantic layer acknowledges that some claims are not cleanly decidable. Here deppy uses oracle-judged predicates with rubrics, prompt templates, content hashing, confidence thresholds, and provenance. The point is not to pretend undecidable meaning can be reduced to SMT, but to keep it typed and inspectable anyway.

PROOF

The proof layer holds Lean-facing theorem material and proof artifacts. It is where formally checkable theorems, residual obligations, and assumption boundaries become visible. Crucially, the proof layer does not erase the distinction between theorem-proved facts and semantically accepted claims.

CODE

The code layer holds executable source, AST-level representations, and runtime-adjacent artifacts. In the hybrid worldview, code is a layer that must agree with the rest, not the final unquestionable source of truth. A code section can still fail to glue with intent, structure, semantics, or proofs.

\[\text{INTENT} < \text{STRUCTURAL} < \text{SEMANTIC} < \text{PROOF} < \text{CODE}\]

One useful reading is to say that the plain type system answers the question “what formal object are we talking about?”, while the hybrid type system answers the question “across which layers has that object been specified, checked, justified, and implemented?”. In that sense the hybrid layer is epistemic structure attached to ordinary type-theoretic content.

Another reading is computational: the plain system is representation and local reasoning, whereas the hybrid system is orchestration and evidence management. The former defines what a dependent pair or refinement means to the repository. The latter manages how such objects travel through generation, solver discharge, oracle judgment, and proof translation.

Relationship

How plain and hybrid relate structurally

The cleanest way to understand deppy is as a stack. The plain system gives the project its language of types, refinements, binders, and symbolic manipulation. The hybrid system then interprets that language across a product site that includes intent, structural facts, semantic judgments, proofs, and code. When the repository says hybrid dependent types, it does not mean the plain system vanished; it means the plain type language has been lifted into a trust-aware, multi-layer semantics.

That stacking explains the code organization. The plain modules live in `src/deppy/types/` and behave like a conventional foundational library of type expressions and proof-relevant operations. The hybrid modules in `src/deppy/hybrid/` repeatedly import, mirror, extend, or compile back into those plain forms. Oracle discipline

Controlled oracles and the split between structural and semantic facts

The hybrid system's most consequential distinction is between structural predicates and semantic predicates. Structural predicates are meant to be decidable or at least deterministically checkable. Semantic predicates represent claims whose meaningful evaluation is not reducible to the same machinery. Instead of collapsing the two, deppy packages semantic judgments into the oracle monad with explicit trust, confidence, provenance, caching, and retry behavior.

This move prevents a common failure mode in LLM-heavy systems: treating every generated statement as equally trustworthy once it appears in a formal-looking pipeline. In deppy, solver-backed facts and semantic judgments are both typed, but they are not the same kind of thing. The semantic layer keeps its own evidence structure and can later be translated, cached, summarized, or demoted without polluting the rest of the system.

It also explains the project's anti-hallucination posture. Structural hallucinations are failures of structural inhabitation: nonexistent APIs, impossible signatures, or missing fields. Semantic hallucinations are failures of meaning: outputs that do not actually satisfy the claimed behavior even if they type-check structurally. The split is therefore not merely philosophical; it directly powers how the repo classifies generated mistakes.

\[\varphi = \varphi_d \land \varphi_s,\qquad \varphi_d \in \mathrm{SMT/runtime},\ \varphi_s \in \mathrm{oracle/semantic}.\]

A third reading is ergonomic: the plain system would be hard for many users to encounter directly in day-to-day work, because it is deep infrastructure. The hybrid system, mixed-mode surface syntax, and site-level explanations are what make the mathematics accessible in practical workflows. Yet the accessibility layer keeps folding back into the plain core every time a decorator turns into a type obligation or a synthesis hole turns into a witness-bearing object.

A final reading is research-facing. deppy wants one mathematics that can describe bug localization, module gluing, generation planning, hallucination detection, and proof discharge. The plain type system is necessary but insufficient for that ambition. The hybrid system is the repo's answer to insufficiency: keep the rigorous core, then add layers, trust, and cohomology-aware glue conditions until the same language can describe much more of the workflow.

Local to global

Cohomology and gluing: why the repository talks this way

The project repeatedly describes bugs, incompatibilities, or failed integrations as cohomological obstructions. The most charitable and precise way to explain that choice is to note that many software workflows are genuinely local-to-global: separate modules, stages, or verification layers can each look acceptable while the total artifact still fails to glue. The layered presheaf and Cech-complex story provide a principled language for that failure.

In a simple reading, local sections are type assignments, partial contracts, or layer-specific artifacts attached to overlapping regions of a program or plan. Restriction maps describe how a richer or higher layer projects to a lower or adjacent one. If all of those local pieces match on overlaps, they should descend to a global section. If they do not, the mismatch can be captured as an obstruction class. That is the core mathematical picture.

The page should still be careful. The repository sometimes speaks in a dramatic style, and not every practical deppy run literally computes a deep symbolic spectral sequence. But the language is not empty. The implemented algebraic-geometry module, the obstruction localizer, equivalence descent code, and the hybrid type docstrings all show that the local-to-global story is a real organizing principle, not just branding.

Precision notes

Caveats, duplicates, and places to stay precise

Caveat 1

The repo contains both a lightweight `HybridTrustLevel` in `deppy.hybrid.__init__` and a richer trust lattice in `deppy.hybrid.core.types` / `core.trust`. A precise explanation should say these are overlapping public convenience surfaces rather than one single canonical definition in one place.

Caveat 2

The project often speaks as though every semantic predicate lives cleanly inside theorem-prover territory. In practice the Lean translation pipeline explicitly needs axioms or residual assumptions for some semantic claims, so the page should distinguish theorem checking from trust-labeled semantic acceptance.

Caveat 3

The phrase “bugs are H¹ classes” is part of the repo's core self-description, but it should be explained operationally: the point is that local compatibility does not imply global gluing, and the cohomology machinery names that failure. It should not be written as though every ordinary bug report is literally computed from a full symbolic Cech complex in all workflows.

Caveat 4

The mixed-mode layer makes very strong claims about natural-language specifications, but its own design invariant is graceful degradation before compilation. That means some user-visible surface forms are placeholders or pass-throughs until later stages translate them into obligations.

Caveat 5

The plain system is not “old and replaced” while the hybrid system is “new and real”. The repo keeps relying on the plain system for substitution, refinement syntax, dependent formers, and type-level structure. Hybrid extends it; it does not erase it.

Caveat 6

Likewise, the hybrid layer is not only a verification add-on. It affects planning, type expansion, diagnostics, generation, and anti-hallucination. Treating it as “the theorem prover part” would undersell what the repository is actually doing.

These caveats are not a criticism of the project so much as a guide for writing an exact page. The repo is ambitious and occasionally says the same thing at different abstraction levels or with different public convenience surfaces. A careful explanation should make those duplications legible instead of pretending the tree is flatter than it is.
